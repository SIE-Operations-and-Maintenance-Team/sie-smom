为部门开发人员按统一风格、形式编码提供一个设计和编码标准，从而改进代码的可读性，可维护性，提高代码质量从管理的角度，制定的标准、规范是必须要严格遵守和执行的，不能随意的篡改或者事实上违背组织已经采用的标准和规范。因此，程序员在编码过程中应当完全遵守该规范的所有细则，如果在特定的情况下用某种与该规范细则相冲突的形式更有利于程序的可读性，可以考虑保留这种形式，但必须是在经过正式评审、严格控制的情况下实施。

1.2.3. 源代码管理规范
1. 部门项目源代码由刘耀辉和林谣明管理
2. 任何源代码文件包括设计文档等技术资料不得涉外网络环境形式进行传输
3. 代码签入时必须获取最新代码，解决冲突，再次获取最新代码，确认无冲突，编译通过后才可签入。
4. 签入代码时加上注释，简要注明修改的内容
5. 签入时务必与上一版本进行比较，避免将别人的代码覆盖掉
6. 数据库脚本统一放在TFS上管理

1.2.4. 类图设计规范
1. 统一使用ModelFirst进行类图设计
2. 一个类图只包含一个领域实体（一个根实体及其子实体和一个层级的相关引用），不同类型按颜色区分,如：


3. BS开发中基类不能包含列表属性，不然生成界面解析异常
4. 枚举必须指定值；是否非空要体现在关联线上
5. 数据库表名，模块名_表释义名，如果主表已代表整个模块，则用全大写字母构成表名，如工单表表名WO，工单BOM表表名 WO_BOM,工序BOM表表名 WO_PROC_BOM
6. 数据库表名长度，Oracle标识符最大长度为30，生产数据库表同时会生成序列和同步序列，故数据库表名尽量控制在15个字符以内，过长的表名可以使用缩写例如：实体DefectResponsibilityCategory  表名：DEF_RESP_CATE
7. 数据库表名定义：不能包含.和关键字
8. 类名和类成员名都采用Pascal命名法，并添加正确的说明，即每个单词首字母大写，如类名：Employee，EmployeeGroup，属性名：Code，BeginTime，IsOverDay
9. 如果属性名不是全大写，则不允许相连的两个字母大写，如工单类型，不能命名成WOType,此时数据库生成的字段会变成W_O_TYPE 而是WoType,数据库生成的字段是WO_TYPE
10. 某类的属性名通用的词不用加其它修饰词，如：Code、Name、Description、Type 等，状态统一用State，不用Status
11. 类的属性只建业务相关的属性名，业务属性名不和框架属性冲突，如不能包含Id，CreateBy，UpdateBy，CreateDate，UpdateDate，InvOrgId, IsPhantom, SyncId否则生成会异常（Id是主键，创建人，修改人，创建时间和修改时间在继承的DataEntity中已经存在，InvOrgId, IsPhantom, SyncId为框架相关字段）；
12. 建类名和属性名必须要把备注加上，生成类和属性的说明会根据备注自动生成

13. ModelFirst建的类的层级最好跟项目工程的结构一致，这样生成的类和相关文件夹拷贝到工程中不需要修改命名空间
14. 类的名称空间为“对应工程名.文件夹名称（文件夹以s结尾）”，如：namespace SIE.Demo.Items（SIE.Demo为工程名，Items为文件夹名称）
15. ModelFirst建关系时我们只建一般关系(一对一的关系)，组合关系（一对多的关系），一般关系和组合关系的属性名不要与普通属性名冲突

16. 多对多的关系不要在ModelFirst用关系去体现，通过中间表去处理；

1.2.5. SMOM项目规范

所有名称不要使用汉语拼音，而使用英文，标识符应当直观，尽量做到可望文知意。
1.2.5.1. 项目解决方案命名
可以体现客户名称和所属项目领域名称 如：SMES.HisenseWMS。
1.2.5.2. 工程的命名及搭建
体现模块名，服务端和客户端分开工程放置，如SIE.Inventory/SIE.Web.Inventory。
工程搭建：原则上同一业务类型搭建一对工程，如仓库相关业务，搭建工程为SIE. Warehouse/SIE.Web.Warehouse;切忌不要一个功能搭建一个工程(这样比较难维护)
1.2.5.3. 类命名规范通用说明
1.	命名空间名称采用Pascal规则，即首字母要大写；
2.	命名空间名称尽量反映其内容所提供的整体功能。如：namespace SIE.Defects
3.	文件名使用Pascal规则命名，即首字母要大写；
4.	文件名使用能够反映功能的名词或名词短语命名；
5.	不要使用“I”、“C”、“_”等特定含义前缀；
6.	类的注释使用“///”三斜线注释。
1.2.5.4. 数据提供者类命名规范
1.	必须包含命名空间，且命名空间为对应工程名，如工程名为SIE.Test，则命名空间为：namespace SIE.Test；
2. 	类的命名规范：XXXDataProvider，即以DataProvider结尾；
3.	工程与工程之间的数据提供者类命名唯一，建议以“工程名+DataProvider”进行命名，如工程为SIE.Test，则命名为TestDataProvider。
1.2.5.5. 模块初始化类
1.	必须包含命名空间，且命名空间为对应工程名，如工程名为SIE.Test，则命名空间为：namespace SIE.Test；
2.	类的命名规范：XXXModule，即以Module结尾。 
1.2.5.6. 实体类
1.	实体类必须建在服务端工程中；
2.	实体类必须包含在对应文件夹下；文件夹以“实体名+S”进行命名；
3.	实体类必须包含命名空间，且命名空间为“对应工程名+文件夹名”；
4.	实体类为部分类
5.	实体类继承的父实体不能包含子列表属性；
6.	实体类必须标记特性[RootEntity/ChildEntity, Serializable]；
7.	实体类中只包含对应的属性，不要去写与业务逻辑相关的代码；
8.	实体类中的实体配置以“类名+Config”进行命名。
1.2.5.7. 查询实体类
1.	查询实体类必须建在服务端工程中；
2.	类的命名以“实体名+Criteria”进行命名；
3.	必须包含命名空间，且命名空间为“对应工程名+文件夹名”。
1.2.5.8. ViewModel类
1.	类的命名规范：XXXViewModel，即以ViewModel结尾。
1.2.5.9. 属性
1.	属性名命名规范，使用Pascal规则命名，即首单词大写；
2.	相邻两个字母不能同时大写；如WO，映射的数据库字段为W_O；Wo映射的数据库字段为WO；
3.	属性名不能包含关键字；
4.	定义的属性名首字母不能小写，如果首字母为小写，映射不了数据库字段；
5.	属性名不能冲突(常见冲突为与框架属性冲突)，属性名冲突会造成映射数据库失败，而且失败框架没有具体的异常提示；
6.	实体新增的属性尽量用代码段添加，不要复制；如果是从其他实体中复制过来的属性，属性关联的实体一定要记得修改，否则会抛异常；

7.	属性的Label标签必须标记；
8.	String类型的数据默认映射数据库表字段的长度为80，验证输入的长度为20，要调整为其他长度控制，需要自己额外去设置；
1.2.5.10. 控制器类
1.	控制器类必须建在服务端工程中；
2.	类的命名规范：XXXController,即以Controller结尾；
3.	控制器中非私有方法必须是虚方法；
4.	控制器类中访问修饰符或功能相同的方法应该放在一起； 
5.	一个实体对应一个控制器，操作对应实体的方法在对应控制器中去实现；如实体名为Item，则控制器的名称为ItemController,对Item进行增删改查方法的实现在ItemController中进行实现，其他地方进行调用。
1.2.5.11. 规则类
1.	规则类必须建在服务端工程中；
2.	类的命名规范：XXXRule，即以Rule结尾。

1.2.5.12. 附件类
1.	附件类的命名规范：XXXAttachment；
2.	附件作为子实体展示，标签的特性为：[ChildEntity, Serializable]；
3.	附件的实体配置需要启用实体鉴别器，通过Meta.EnableDiscriminator(“XXX”)进行启用设置;
4.	使用附件需要定义数据仓库类，仓库类的命名规范：附件类+Repository；
5.	附件仓库类命名不能与其他附件仓库类命名冲突，否则会出现上传的附件地址信息为空，附件没有上传成功。
1.2.5.13. 打印类
1.	标签打印类的命名规范：XXXLabelPrintable；
2.	单据打印类的命名规范：XXXPrintable；
3.	标签打印和单据打印必须标记特性[DisplayName("打印名称")]。
1.2.5.14. 配置项类
1.	框架通用配置项配置编码自动生成：[EntityWithConfig(typeof(NoConfig))]
2.	配置项实体的命名规范：必须以XXXConfigValue命名，最后必须带“Value”
3.	规则生成说明类命名规范：必须是配置项的类名“Value”之前的部分，如配置项的类名为“XXXConfigValue”时规则生成说明类必须命名为“XXXConfig”。
1.2.5.15. 界面配置类
1.	类名建在界面端工程中；
2.	必须包含名称空间，名称空间为“对应客户端工程名.文件夹名”；
3.	类的命名规范：XXXViewConfig，即以ViewConfig结尾；
4.	在ConfigView方法中不要去配置命令，如下配置了命令会导致所有视图配置方法都会加对应命令

5.	界面配置编辑模式和命令，有先后依赖顺序，先配置编辑模式，再配置命令，如下：

6.	自定义分组名，使用常量定义分组名，不要使用字符串去设置分组；





1.2.5.16. 命令类
1.	命令类的文件必须放到Commands文件夹下；

2.	命令文件的命名必须以Command结尾；
3.	命令的JS文件需要嵌入到资源（所有js文件都需要嵌入到资源）

4.	重写的命令一定要加meta，并且不能换行

5.	如果命令前后端有数据交互，同事需要是个js和cs文件，且两个文件的全命名空间要完全一致，如：
	Js命令文件命名：SIE.Web.Test.UnitTests.Commands.AddUnitCommand

	CS命令文件的名称空间和文件名：SIE.Web.Test.UnitTests.Commands和AddUnitCommand，与js的命名空间一致。 

6.	与命名无关的文件不能放在Commands文件夹下；
1.2.5.17. 调度类
1.	调度任务类命名规范：XXXJob，即以Job结尾；
2.	调度任务类必须标记Job标签，用于告诉系统当前的类是一个调度任务类；
3.	调度的执行方法ExecuteJob中，必须要记录日志，通过AddLog进行日志的记录；
4.	自定义调度参数类命名规范：XXXJobParameter，即以JobParameter结尾。
1.2.5.18. 预警类
1.	预警任务类命名规范：XXXAlert，即以Alert结尾；
2.	预警配置类命名规范：XXXAlertConfig，即以AlertConfig结尾；
3.	预警参数类(返回结果类)命名规范：XXXAlertResult，即以AlertResult结尾。
1.2.5.19. 推送类
1.	推送插件类命名规范：XXXSender，即以Sender结尾；
2.	推送配置类命名规范：XXXSenderConfig，即以SenderConfig结尾。
1.2.5.20. 其他规范
1.	方法名采用Pascal规则，首字母大写；
3.	方法的参数和变量的命名规范：首字母小写；
4.	方法要有对应的注释，能简要说明方法的逻辑，比较好理解的代码逻辑不需要加注释，方法使用“///”三斜线注释 ；
5.	全局变量命名规范：首字符大写；
6.	方法、变量和参数的命名要通俗易懂，最好能涵盖代码的逻辑，根据名称能知代码的含义； 
7.	每个方法的源程序行数原则上应该少于100行；
8.	常量的命名规范：首字符大写；
9.	静态字段或属性采用Pascal规则，即首字符大写；
10.	枚举和枚举的成员采用Pascal规则，且枚举值遵循从小到大顺序定义；
11.	接口定义使用Pascal规则，且必须以大写“I”开头；
12.	接口名称要有意义，中间不要有下划线“_”等字符；
13.	如果类实现了接口，名称尽量和接口相同， 只是省掉“I”字符；
14.	不允许变量名、类名、属性名、方法名等与系统标识符重名；
15.	代码行文注释采用“//”和“/**/”进行，应该尽量说明问题。
16.	语句嵌套层次尽量控制在四层之内，且尽量避免代码重复，能复用则复用；
17.	SQL语句全部大写，对较为复杂的SQL语句加上注释，说明其功能；
18.	SQL连接符OR、IN、AND、以及=、<=、>=等前后加空格；
19.	SQL语句使用明确的列代替 SELECT *；
